# Selecting data using KQL

Here is the detailed architectural breakdown note on selecting columns, creating calculated fields, and understanding the linear processing model of **KQL (Kusto Query Language)**, structured for your DP-600 exam preparation.

## 1. Navigating KQL Query Sets

To begin writing KQL scripts inside Microsoft Fabric's Real-Time Intelligence workspace, you interact directly with a **KQL Query Set** artifact.

- Expanding your target KQL database node in the Explorer pane maps out all background metadata schemas, including physical **Tables**, **Shortcuts**, **Materialized Views**, and **Functions**.
- _Exam Tip:_ For the DP-600 certification curriculum, the practical focus is strictly bound to querying base **Tables**. Advanced objects like Materialized Views and relational functions map heavily to the DP-700 data engineering pathway.

## 2. Core KQL Table Referencing & The Pipe Operator (`|`)

The absolute simplest, valid KQL query consists of nothing more than the raw name of the table itself:

```SQL
weather
```

### Escaping Hyphens and Symbols

If an ingestion pipeline imports a table containing special characters, non-standard symbols, or hyphens (e.g., `storm-events`), referencing it nakedly will cause syntax compilation errors. You must explicitly escape the database asset name using an array bracket and single quote pattern:

```SQL
-- Correctly escaping tables with hyphens
['storm-events']
```

### The Pipeline Architecture Model

Unlike relational SQL queries—which are bound to a rigid, fixed clause evaluation path—KQL operates as an open-ended, linear **Pipeline**.

Each independent command block is separated by a **Pipe Symbol (`|`)**. The pipe operator takes the tabular dataset output generated by the preceding step, streams it downstream, and passes it into the next sequential evaluation handler block.

## 3. Projecting and Renaming Columns: The `project` Operator

The KQL **`project`** operator is the structural equivalent of the `SELECT` clause in T-SQL or PySpark. It dictates exactly which columns pass vertically through the active query pipeline to the final output pane, filtering out unneeded fields.

### Explicit Column Projection Syntax

```SQL
weather
| project State, EpisodeId, StartTime, EndTime, EventNarrative
```

### The Inverted Aliasing Pattern

When renaming a column during the projection phase, KQL enforces an **inverted layout** compared to standard SQL. Instead of writing `[Old Column] AS [New Name]`, KQL strictly requires declaring the **New Name first**, followed by a single equal sign, followed by the original column variable:

Code snippet

```SQL
-- KQL Aliasing Syntax: [New Name] = [Old Column]
weather
| project State, StartDate = StartTime, EndTime
```

## 4. Calculated Fields: `project` vs. `extend`

Adding calculated metrics highlights the unique linear dependency model of KQL pipelines.

### The Immediate Dependency Trap

You cannot reference a newly minted column alias inside the exact same operator statement block where it is being defined. For instance, creating an alias `StartDate = StartTime` and simultaneously attempting to compute `EndTime - StartDate` on the same line will crash the execution engine. One calculation must physically complete before downstream operators can recognize it.

### Using the `extend` Operator

To append brand-new calculated columns to your dataset while safely maintaining all previously existing fields, deploy the **`extend`** operator down the pipeline stream:

```SQL
weather
-- Step 1: Filter columns and define a clean alias
| project State, StartDate = StartTime, EndTime
-- Step 2: The pipe pushes the mutated dataset down to a fresh execution boundary
| extend Duration = EndTime - StartDate
```

### Structural Comparison Matrix

| **KQL Operator** | **Target Action on Schema**           | **SQL Syntax Equivalent**    | **Impact on Unselected Columns**                         |
| ---------------- | ------------------------------------- | ---------------------------- | -------------------------------------------------------- |
| **`project`**    | Selects and filters specific columns  | `SELECT column1, column2`    | **Deletes** all other unnamed columns from the pipeline. |
| **`extend`**     | Appends a brand-new calculated column | `SELECT *, (Calc) AS NewCol` | **Preserves** 100% of existing columns in the stream.    |

# Further selecting columns and ordering data using KQL

## 1. Sorting Mechanics and the Default Casing Trap

To sort the row output of an event stream in KQL, you deploy either the **`order by`** or **`sort by`** operator. They are completely interchangeable.

> **Exam Tip / Architectural Trap:** In standard SQL dialects (T-SQL, Spark SQL), writing a basic `ORDER BY column` defaults to an ascending sort (A to Z). In **KQL**, the execution engine reverses this logic: if you do not explicitly pass a direction modifier, **KQL defaults to a descending sort (Z to A / Largest to Smallest)**.

```sql
-- KQL Default Behavior: Sorts descending automatically
weather
| project State
| order by State

-- Enforcing an Ascending Sort explicitly
weather
| project State
| order by State asc
```

### Explicit Null Placement

When handling missing data points (`null`), KQL allows you to strictly control where unknown rows cluster relative to your sorted indices by adding **`nulls first`** or **`nulls last`** to your sorting line:

```sql
weather
| sort by State asc nulls last
```

## 2. Advanced Projection Operators: Modifying Schemas

While the standard `project` operator filters a table down to _only_ the named columns, KQL includes specific target project variations to handle data modifications efficiently:

### A. The `project-away` Operator (Drop Columns)

If you have a massive table containing dozens of columns and you only want to remove one or two non-essential fields (or drop an interim calculation column to clean up your network pipeline), use **`project-away`**. It explicitly drops designated columns while automatically preserving all other variables.

```sql
weather
| project-away EventNarrative
```

### B. The `project-rename` Operator (Clean Aliasing)

If you simply want to rename a column header without filtering out any other data from the pipeline stream, use **`project-rename`**. It follows the standard KQL inverted aliasing pattern: `[New Name] = [Old Name]`.

```sql
weather
| project-rename Narrative = EpisodeNarrative
```

### C. Overwriting Columns with `extend`

If you deploy the `extend` operator and pass a variable name that **already exists** in the upstream data stream, the engine will not create a duplicate column or append a numeric suffix (like `column1`). Instead, it dynamically **overwrites and replaces** the content of the existing column.

```SQL
weather
-- Overwrites all existing strings inside EpisodeNarrative with the static text literal "Hi"
| extend EpisodeNarrative = "Hi"
```

## 3. Query Whitespace and Commenting Rules

Because KQL query sets execute block instructions interactively, understanding how the parser treats code layout prevents broken production deployments.

- **Single-Line Comments (`//`):** To mute a pipeline step or write developer notes, preface the text with two forward slashes. Any text following `//` on that line turns green and is bypassed by the engine. It can be placed at the start of a line or mid-line after a command.
- **The Empty Line Trap:** A completely blank, un-piped space inside a query block breaks the continuous chain. The parser interprets an empty line as a **hard terminator boundary**, detaching the lower code fragments, draining their color syntax, and throwing an `Incomplete Fragment` exception if executed.
- **The Fix:** If you need to inject vertical spacing to improve code legibility, always fill the empty lines with a commented pipeline indicator character string (`//`).

```SQL
-- CORRECT Layout with spacing:
weather
| project State, StartTime
//
| order by State asc
```

## Technical Summary Matrix: KQL Column Manipulators

| **KQL Operator**     | **Core Purpose**                             | **Effect on Unnamed Fields** | **SQL Structural Match**    |
| -------------------- | -------------------------------------------- | ---------------------------- | --------------------------- |
| **`project`**        | Retains only listed columns.                 | **Deletes them**             | `SELECT col1, col2`         |
| **`project-away`**   | Explicitly drops a few target columns.       | **Preserves them**           | `ALTER TABLE DROP COLUMN`   |
| **`project-rename`** | Renames a header without filtering data.     | **Preserves them**           | `SELECT *, col AS new_name` |
| **`extend`**         | Appends new items or overwrites old entries. | **Preserves them**           | `SELECT *, (Calc)`          |

# Limiting the number of rows

## 1. Horizontal Row Restraints: `take`, `limit`, and `sample`

When evaluating massive telemetry tables, streaming every row into your presentation layer degrades network and memory performance. To quickly sample horizontal layers of data, KQL provides three distinct filtering operators:

```sql
DATA INPUT: weather (59,066 Rows)
   │
   ├──> | take 10  ──────> Pulls the absolute first 10 rows sequentially (Fast)
   │
   └──> | sample 10 ─────> Pulls 10 rows without relying on absolute sequence order (Optimized for big data scans)
```

- **`take [n]`:** Pulls the absolute first `[n]` records from the sequential structure of the dataset. It is highly efficient for checking row composition layouts.
- **`limit [n]`:** This is an exact functional **synonym** for `take`. Writing `limit 10` or `take 10` compiles down to the identical backend engine execution tree.
- **`sample [n]`:** Retrieves `[n]` records, but **violates sequential processing guarantees**. Instead of forcing the engine to read chronologically from the top slice of the table, it pulls rows non-sequentially. This delivers massive performance gains when running rapid diagnostic inspections across vast, multi-billion-row columnar databases.

## 2. Extracting Unique Dimensions: `distinct` vs. `sample-distinct`

### The `distinct` Operator

To eliminate duplicate rows and extract an audited list of unique values across a column dimension (such as isolating unique region names), use the **`distinct`** operator. It functions exactly like the `SELECT DISTINCT` modifier in relational SQL:

```sql
-- Isolates the unique occurrences within the 'State' column, returning 67 rows
weather
| distinct State
```

### The `sample-distinct` Operator

If you need to extract a small unique subset of categories from a massive table, appending a standard `limit 10` down the pipe after a `distinct` statement forces the engine to parse the _entire_ table first to deduplicate it, creating a processing bottleneck.

To bypass this overhead, use **`sample-distinct`**. It commands the engine to immediately stop searching once it scans and satisfies your requested sample subset count:

```sql
-- Syntax: sample-distinct [Count] of [ColumnName]
weather
| sample-distinct 10 of State
```

## 3. Top-N Bounds Scans: The `top` Operator

To isolate the extreme boundaries of a dataset based on an ordered metric sequence (such as identifying the absolute highest values), use the **`top`** operator.

`top` combines the horizontal filtering of `take` with the sorting mechanics of `sort by` inside a single execution block.

```SQL
-- Syntax: top [Count] by [ColumnName] [asc/desc]
weather
| top 10 by State asc nulls last
```

- **KQL Order Default Reminder:** Just like standard `sort by`, the `top` operator defaults to a **descending sequence** unless you explicitly suffix the column using the **`asc`** modifier keyword.

## 4. Multi-Query Script Boundaries (The Semicolon Rule)

KQL query sets support authoring multiple disconnected analysis statements within a single design canvas tab window.

- **The Execution Barrier:** If you have two independent pipeline blocks written on a page separated by white space, running the entire script canvas simultaneously will fail with code fragment errors unless you explicitly instruct the parser where statements end.
- **The Semicolon Requirement (`;`):** To execute a multi-query batch that renders multiple separate data tables in your results window concurrently, **you must terminate every query in the chain (except the absolute final statement) with a semicolon**.

```SQL
-- Query Statement 1 (Must end with semicolon)
weather | take 5;

-- Query Statement 2 (Final block doesn't require termination)
weather | distinct State | top 5 by State asc
```

## KQL Row Limitation Operator Matrix

| **Operator Syntax**   | **Returns Sorted Order?**        | **Deduplicates Rows?** | **Primary Performance Goal**                             |
| --------------------- | -------------------------------- | ---------------------- | -------------------------------------------------------- |
| **`take` / `limit`**  | No (Pulls raw data)              | No                     | Fast structural inspection of top data frames.           |
| **`sample`**          | No (Pulls non-sequential data)   | No                     | Maximum performance scanning across big-data sets.       |
| **`distinct`**        | No (Pulls random sequence)       | **Yes**                | Compresses rows to map unique attribute categories.      |
| **`sample-distinct`** | No (Pulls random sequence)       | **Yes**                | Rapidly isolates a limited sample list of unique values. |
| **`top ... by`**      | **Yes** (Defaults to Descending) | No                     | Captures maximum or minimum extreme rows instantly.      |

# Creating a string literal

## 1. Direct Execution Debugging: The `print` Operator

When validating text processing logic or testing string transformations, you do not need to read from a massive physical table. KQL provides a built-in diagnostic tool called the **`print`** operator.

`print` evaluates a scalar expression or static literal value and immediately outputs it as a temporary single-cell row grid.

```sql
-- Basic evaluation rendering a column named 'Result' containing the string
print "Hello World"

-- Defining an explicit column header name for the printed result
print Calc = 1 + 1
```

## 2. Standard String Literals and Enclosure Rules

A standard string literal is an array of alphanumeric characters wrapped within boundary tokens. KQL treats **Single Quotes (`'`)** and **Double Quotes (`"`)** with absolute functional equality.

### Resolving Nested Quotes

This dual-enclosure architecture allows you to easily store strings that contain quote characters as part of their text value without breaking the query compiler:

- **To embed double quotes:** Wrap the entire outer boundary of the string literal in single quotes.

```sql
    print Result = 'She said "Hello" to the team.'
```

- **To embed single quotes:** Wrap the entire outer boundary of the string literal in double quotes.

  ```
  print Result = "The developer's query executed successfully."
  ```

## 3. The Backslash (`\`) and Verbatim Literals

The backslash character functions natively as an **Escape Token**. It modifies the behavior of the immediate character that follows it to inject formatting or system control sequences into a string:

- `\t` $\rightarrow$ Injects a horizontal Tab space.
- `\n` $\rightarrow$ Injects a hard New Line break.
- `\uXXXX` $\rightarrow$ Injects a specific **Unicode** character map sequence utilizing a 4-digit hexadecimal address (e.g., `\u0021` outputs an exclamation mark `!`).

### Bypassing the Escape Engine

If you try to output a single raw backslash inside a standard string (such as defining a file system path like `C:\Data`), the engine flags it as a special character command and causes a compilation syntax error. You can bypass this using two distinct approaches:

#### Approach A: Double Backslash Doubling

You can escape the backslash itself by writing two of them side-by-side (`\\`). The engine evaluates the pair and outputs a single text backslash.

```sql
print Path = "C:\\Data\\SubFolder"
```

#### Approach B: Verbatim String Literals (`@`)

Pre-fixing the string boundary token with an **At symbol (`@`)** activates a Verbatim configuration. This instructs the KQL parser to completely ignore all backslash escape rules, treating every individual backslash strictly as a literal text character.

```sql
-- The @ symbol turns off escape translation rules globally for this string
print Path = @"C:\Data\SubFolder"
```

## 4. Structuring Multi-Line Text: Verbatim Backticks

Standard single and double quotes are strictly limited to capturing text on a single linear line. If a data validation process requires compiling blocks of text spanning across multiple vertical carriage returns, you must implement **Verbatim Multi-Line String Literals**.

````
print MultiLineText = ```
    Line 1: Ingestion Process Initiated.
    Line 2: Target File Pathway: C:\Data\Logs
    Line 3: Operations Completed.
````

````

### Syntax Construction Rules
* A multi-line string must begin with a cluster of **three consecutive backticks (\`\`\`)**—the backward-slanted apostrophe key typically located in the top-left corner of standard keyboards.
* It must terminate with a matching cluster of **three consecutive backticks (\`\`\`)**.
* Multi-line backtick strings automatically operate under Verbatim rules; backslashes inside them are treated as standard literal text without requiring an additional `@` character.

## String Literal Syntax Reference

| String Type Definition  | Opening/Closing Tokens | Escape Rules Applied?          | Best Used For                                                                |
| :---------------------- | :--------------------- | :----------------------------- | :--------------------------------------------------------------------------- |
| **Standard String**     | `'...'` or `"..."`     | **Yes** (`\t`, `\n` processed) | Basic alphanumeric variables and category matching.                          |
| **Verbatim String**     | `@'...'` or `@"..."`   | **No** (Ignores escape rules)  | Hardcoded file paths, regex text patterns, and network URLs.                 |
| **Multi-Line Verbatim** | ` ```...``` `          | **No** (Ignores escape rules)  | Multi-line documentation notes, long text narratives, or raw configurations. |
# Filtering for the entirety of a string

## 1. Syntax Mechanics: The `where` (or `filter`) Operator

To filter individual records out of an active data stream based on explicit conditional criteria, deploy the **`where`** operator.

- **The Functional Synonym:** In KQL, **`where`** and **`filter`** are completely identical synonyms. Writing `| where ...` or `| filter ...` compiles down to the exact same engine execution paths.
- **Case-Sensitivity Guardrail:** By default, KQL is **strictly case-sensitive**. If a data column houses title-case entries (like `"Heavy Rain"`) and your query checks against lowercase values (like `"heavy rain"`), the query returns exactly zero rows.

## 2. Equality and Case-Sensitivity Matrix

KQL uses specific operator modifiers to control text matching behaviors and processing speeds.
### True Equality (`==`)

Requires two consecutive equal signs (`==`) to evaluate a logical comparison.

- _Note:_ A single equal sign (`=`) is reserved exclusively for naming column headers or assigning aliases. True equality checks are strictly **case-sensitive**.

### Case-Insensitive Equality (`=~`)

Combines a single equal sign with a **tilde symbol (`~`)** to bypass case checks entirely.

- _Performance Tradeoff:_ `==` performs faster because it runs an exact binary match. `=~` incurs slightly more compute overhead because the engine must normalize string casing behind the scenes.

### Not Equal (`!=` and `!~`)

To isolate everything _except_ a specific category, replace the leading token with an **exclamation mark (`!`)**.

```sql
-- Case-Sensitive True Match
weather | where EventType == "Heavy Rain"

-- Case-Insensitive Match (Finds "HEAVY RAIN", "heavy rain", etc.)
weather | where EventType =~ "heavy rain"

-- Case-Insensitive Exclusion
weather | where EventType !~ "heavy rain"
````

## 3. Set-Based Filtering: `in` vs. `!in`

Instead of writing long, repetitive logic chains like `where EventType == "Heavy Rain" or EventType == "Blizzard"`, you can declare an array of items using the **`in`** operator.

```sql
-- Case-Sensitive Set Check
weather
| where EventType in ("Heavy Rain", "Blizzard")

-- Case-Insensitive Set Check (Suffix the operator with a tilde)
weather
| where EventType in~ ("heavy rain", "blizzard")

-- Negative Set Check (No space allowed between the '!' and 'in')
weather
| where EventType !in ("Heavy Rain", "Blizzard")
```

## 4. Compound Logic Hierarchies: `and` vs. `or`

When combining multiple filter layers, KQL evaluates them using short-circuit boolean logic. All logical connecting keywords (`and`, `or`, `not`) must be typed in **strictly lowercase**.

- **`and` (Intersection):** Requires **both** conditions on either side to evaluate to true for a row to pass.
- **`or` (Union):** Requires **at least one** condition to evaluate to true.

### Prioritizing Evaluation Paths via Parentheses

When blending `and` and `or` logical gates in a single line, you must wrap your choice boundaries inside **parentheses `()`** to enforce the correct order of operations and avoid logic confusion.

```sql
-- Captures events that are EITHER Heavy Rain OR Blizzard, but ONLY inside the state of Texas
weather
| where (EventType == "Heavy Rain" or EventType == "Blizzard") and State == "TEXAS"
```

### Alternative Global Negative Scoping

Instead of rewriting multiple individual operators to clear out a subset, you can wrap a broad condition block inside a parenthesized **`not()`** statement function:

```sql
weather
| where not(EventType == "Heavy Rain" or EventType == "Blizzard")
```

## KQL Row Filtering Operator Quick Reference

| **Operator Syntax** | **Casing Mode**  | **Evaluation Logic**   | **Target Objective**                                      |
| ------------------- | ---------------- | ---------------------- | --------------------------------------------------------- |
| **`==`**            | Case-Sensitive   | Equals                 | High-performance, exact string matching.                  |
| **`!=`**            | Case-Sensitive   | Does Not Equal         | Filters out a specific exact string profile.              |
| **`=~`**            | Case-Insensitive | Equals                 | Safely handles messy, mixed-case user entries.            |
| **`!~`**            | Case-Insensitive | Does Not Equal         | Excludes mixed-case string variants safely.               |
| **`in`**            | Case-Sensitive   | Contained in Array     | Evaluates a value against a fixed hardcoded list.         |
| **`in~`**           | Case-Insensitive | Contained in Array     | Evaluates a value against a list without checking casing. |
| **`!in`**           | Case-Sensitive   | Not Contained in Array | Excludes rows that match any items in the array.          |

# Filtering for part of a string

## 1. Word Tokenization vs. Substring Searching: `has` vs. `contains`

When analyzing free-text log fields (such as `EpisodeNarrative`), the standard true equality operator (`==`) fails unless your filter matches the _entire entry_ exactly. To perform partial string inspections, you must strategically choose between word-level matching (`has`) and character-level matching (`contains`).

### The `has` Operator (Tokenized Word Search)

- **Behavior:** Scans the text for a **complete, standalone word**. It automatically accounts for boundaries such as spaces, commas, periods, or punctuation marks.
- **Performance Impact:** **Highly Recommended.** It is significantly faster than substring matching because the KQL engine optimizes for tokenized word indexes under the hood.

### The `contains` Operator (Substring Search)

- **Behavior:** Scans the text for a **raw sequence of characters**, regardless of whether they form a full word or are buried inside another word.
- **Performance Impact:** Slower. It forces a scan across character arrays. For example, looking for the character string `"coun"` using `contains` returns matches for _"county"_, _"counties"_, _"counter"_, and _"discount"_.

### Case Sensitivity Formatting

By default, both operators are **case-insensitive**. If you require an absolute exact casing match, append the suffix **`_cs`** (Case Sensitive):

```sql
-- Case-Insensitive Word Search (Finds "County", "COUNTY", "county")
where EpisodeNarrative has "county"

-- Case-Sensitive Word Search (Finds ONLY lowercase "county")
where EpisodeNarrative has_cs "county"
```

## 2. Granular Word Placement: Prefixes and Suffixes

When you need to look for patterns bounded at specific edges of a word or an entire string field, deploy specialized positional predicates:

### Word-Level Anchors

- **`has_prefix`:** True if a standalone word **begins** with the specified character string. (e.g., `has_prefix "coun"` flags _"county"_ and _"count"_, but skips _"discount"_).
- **`has_suffix`:** True if a standalone word **ends** with the specified character string.

### String-Field-Level Anchors

- **`startswith`:** True if the **entire text field string** begins with the target pattern.
- **`endswith`:** True if the **entire text field string** ends with the target pattern.

_Note:_ Just like `has` and `contains`, these structural modifiers feature exact case-sensitive variants by appending the `_cs` token (e.g., `startswith_cs`).

## 3. Multi-Word Arrays: `has_any` and `has_all`

Instead of writing repetitive logic blocks separated by multiple `or` or `and` operator paths, KQL provides native vector array functions to search for multiple distinct words simultaneously.

### The `has_any` Operator (The `OR` Array)

Evaluates to true if **at least one** of the words listed in the bracketed array matches a tokenized word inside the target text field.

```sql
-- Shorthand for: has "showers" or has "county"
where EpisodeNarrative has_any ("showers", "county")
```

### The `has_all` Operator (The `AND` Array)

Evaluates to true only if **every single word** listed in the array appears somewhere within the target text field, regardless of their relative sequence position.

```sql
-- Shorthand for: has "showers" and has "county"
where EpisodeNarrative has_all ("showers", "county")
```

## 4. Native Linear Intersection: Multi-`where` Chaining

A major architectural departure from traditional relational T-SQL is KQL's ability to **chain multiple independent `where` statements sequentially** down the data pipe.

In SQL, writing multiple disconnected `WHERE` clauses causes a syntax crash. In KQL, stacking consecutive piped filters is fully supported and acts as a logical **`and` intersection**:

```sql
weather
| project EpisodeNarrative
| where EpisodeNarrative has_prefix "coun"   -- Step 1: Retains rows with words starting with 'coun'
| where EpisodeNarrative !has_prefix "count"  -- Step 2: Filters that subset to exclude 'count' prefixes
| where EpisodeNarrative !has_prefix "couple" -- Step 3: Further filters to exclude 'couple' prefixes
```

- **Engine Processing Flow:** Each row passing through the pipe is filtered at Step 1, the remaining subset is passed into Step 2, and the surviving records are filtered at Step 3. All conditions must be satisfied for a record to reach the final output grid.

## Partial String Operator Reference

| **Operator Syntax** | **Scope Level** | **Matching Requirement**                 | **Performance Profile**      |
| ------------------- | --------------- | ---------------------------------------- | ---------------------------- |
| **`has`**           | Word Level      | Standalone tokenized word                | **Fastest partial match**    |
| **`contains`**      | Character Level | Substring string pattern anywhere        | Slower character scan        |
| **`has_prefix`**    | Word Level      | Start of an individual word              | Highly optimized             |
| **`startswith`**    | Field Level     | Absolute start of the entire text row    | High performance             |
| **`has_any(...)`**  | Word Array      | Matches **any** listed item (OR logic)   | Clean shorthand optimization |
| **`has_all(...)`**  | Word Array      | Matches **all** listed items (AND logic) | Clean shorthand optimization |

# Aggregating data

## 1. Relational Merging: The `summarize` Operator

The KQL **`summarize`** operator is a dual-purpose engine component. It combines the functionality of SQL's separate `SELECT (Aggregates)` and `GROUP BY` clauses into a single execution step.

### Extracting Distinct Lists (The Suffix Mirror)

If you write a `summarize by` block without declaring any explicit calculation metrics, it functions identically to the KQL `distinct` operator, collapsing identical text rows down into unique categories.

```sql
-- Method A: Traditional deduplication
weather | distinct State

-- Method B: Summarize compression (Outputs the exact same 67 rows)
weather | summarize by State
```

## 2. Multi-Metric Aggregations & Custom Aliasing

To compute mathematical summary metrics across clustered dimension categories, pass your calculation functions before the **`by`** keyword. You can declare multiple calculations on the same line separated by commas.

### Essential Aggregation Functions

- **`count()`**: Calculates total row metrics per category cluster. _Note:_ Unlike T-SQL or Spark SQL, do **not** place an asterisk inside the parentheses (`count(*)`). It must be left completely empty: `count()`.
- **`sum()`** / `avg()` / `max()` / `min()`: Evaluates traditional mathematical boundaries across a specific targeted numeric column.
- **`dcount()` (Distinct Count)**: Calculates the unique count of distinct elements in a column (e.g., counting unique event types per state). While the alternative syntax `count_distinct()` exists, **`dcount()` is heavily optimized and recommended for big data performance.**

### Inverted Aliasing Syntax

Just like the `project` operator, assigning a clean column header name to a calculation requires writing the **New Alias Name first**, followed by a single equal sign (`=`), followed by the function:

```sql
-- Structure: summarize NewName = Function() by Dimension1, Dimension2
weather
| summarize
    TotalRows = count(),
    TotalInjuries = sum(InjuriesDirect),
    UniqueEventTypes = dcount(EventType)
by State
```

## 3. High-Performance Conditional Aggregations: The `*if` Suffix

Instead of writing a broad query that filters down an entire table before aggregating, KQL allows you to execute granular, inline evaluations by appending the **`if`** suffix directly onto your core mathematical functions.

```sql
weather
| summarize
    FloodCount = countif(EventType == "Flood"),
    -- Case-Sensitive count if true
    RainInjuries = sumif(InjuriesDirect, EventType =~ "rain")
    -- Case-Insensitive sum if true
by State
```

### Supported Conditional Functions

- **`countif(Predicate)`**: Increments a row counter only when the inner conditional expression evaluates to true.
- **`sumif(Column, Predicate)`**: Appends a target numeric cell value to the running calculation only when the row satisfies the predicate rule.
- **`avgif()` / `maxif()` / `minif()`**: Computes moving mathematical scales targeted strictly at matching record subsets.

## 4. Query Architecture: Where to Place the `where` Operator

Because KQL allows you to position a `where` operator either before or after a `summarize` block, understanding the underlying performance impact on the Fabric capacity engine is a highly tested concept.

```sql
PRE-SUMMARIZE FILTER (High-Performance Engine Optimization):
weather ──> | where State in~ ("TEXAS", "KANSAS") ──> | summarize (Aggregates Only 2 States)

POST-SUMMARIZE FILTER (High Overhead Scan):
weather ──> | summarize (Aggregates ALL 67 Regions) ──> | where State in~ ("TEXAS", "KANSAS")
```

### Path A: Pre-Summarize Filtering (The SQL `WHERE` Mirror)

Placing the `where` step **before** the `summarize` block cuts down data processing early. The engine immediately drops unneeded rows, forcing the `summarize` function to run its calculations on a highly compressed dataset. **This is the recommended development practice.**

### Path B: Post-Summarize Filtering (The SQL `HAVING` Mirror)

Placing the `where` step **after** the `summarize` block forces the engine to aggregate every single row across all 67 states first. Once that massive summary table is built, the downstream filter discards the unwanted rows.

However, post-summarize filtering is **mandatory** when you need to filter based on the output of an aggregation calculation (such as dropping small regions with minimal activity):

```sql
weather
| summarize InjuryVol = sum(InjuriesDirect) by State
| where InjuryVol >= 100 -- Functions exactly like a 'HAVING' clause in relational SQL
```
